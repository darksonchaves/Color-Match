IDENTIDADE DO PROJETO ───────────────────────── Aplicativo web (HTML5/CSS3/JS vanilla) que interpreta cores em tempo real usando a câmera do dispositivo. O sistema prioriza precisão sobre velocidade. Não há backend; roda 100% client-side via GitHub Pages. 2. PILARES FILOSÓFICOS (imutáveis) ─────────────────────────────────── 1. QUALIDADE E PRECISÃO — interpretação correta acima de tudo. 2. FIDELIDADE — resultado confiável e consistente. 3. SISTEMA SE ADAPTA AO USUÁRIO — acessibilidade e flexibilidade. Velocidade é consequência, nunca objetivo. 3. STACK TÉCNICO ──────────────── - Linguagem: JavaScript (ES6+), sem frameworks. - Estilo: CSS3 puro, mobile-first, viewport lockado. - API: getUserMedia (WebRTC), Web Speech API (TTS pt-BR). - Canvas: getImageData para amostragem cirúrgica de pixels. - Deploy: GitHub Pages (arquivo único index.html). 4. ARQUITETURA DE MODOS (implementados e validados) ───────────────────────────────────────────────────── MODO LIVRE: mira fixa no centro da tela. Análise contínua em loop de requestAnimationFrame. O usuário apenas aponta e lê o resultado. MODO ESTÁTICO: usuário toca na tela para definir um ponto de análise. O sistema captura, preenche o buffer de estabilidade e TRAVA a leitura. Somente o botão "Resetar Estático" libera uma nova análise. Isso evita oscilação por tremor de mão ou mudança de luz. MODO COMPARAÇÃO: captura dois pontos independentes (REFERÊNCIA e AMOSTRA) e exibe um percentual de compatibilidade espectral. Possui painel inferior próprio com thumbnails das capturas, nome das cores e % central. MODO UNIVERSAL (toggle independente): quando ativo, suprime os adjetivos "Claro" e "Escuro", retornando apenas o nome base da cor (ex: "Azul"). Quando inativo (Modo Normal), adjetivos são aplicados por thresholds de V. 5. SISTEMA DE ESTABILIDADE (validado e operacional) ───────────────────────────────────────────────────── - Buffer circular de 8 frames (BUFFER_SIZE = 8). - Mínimo de 7 frames consecutivos para alterar o nome da cor exibido. - Histerese de 6 graus (HYSTERESIS_THRESHOLD = 6) no eixo Hue. - Threshold de mudança de objeto por distância RGB = 55. - Filtro de outliers por mediana dos valores V no buffer. - Cálculo de Hue por média vetorial (atan2 de senos/cossenos) para evitar wrap-around (ex: 359° → 1°). 6. CALIBRAÇÃO PRO (unificada, sempre 30 frames) ─────────────────────────────────────────────── Calibração de BRANCO e PRETO independentes. Cada uma amostra 30 frames em ~1 segundo, filtra outliers afastados da mediana de V, e calcula RGB/HSV médio via média vetorial de H. Após calibração: - Detecta temperatura de cor (fria/quente/neutra) automaticamente. - Aplica compensação de temperatura de cor em todas as amostras futuras. - Normaliza o eixo V (Value) usando a régua branco-preto quando ambos calibrados. - Status exibido no topo: "Base: não calibrada", "Branco V:XX%", "Preto V:XX%" ou "Régua Completa (XX% range)". 7. ÁUDIO / TTS ────────────── - Botão independente "Áudio" (on/off). - Só fala após 600ms de estabilidade (STABLE_SPEECH_MS). - Não repete a mesma cor em sequência, a menos que um reset ocorra. - Cancela fila de fala anterior antes de nova utterance. - Idioma: pt-BR, rate 1.1. 8. MODO COMPARAÇÃO — LÓGICA ESPECTRAL v2.7 (validada) ────────────────────────────────────────────────────── Ordem espectral circular fixa: Vermelho → Rosa → Magenta → Roxo → Azul → Ciano → Verde → Amarelo → Laranja Regras de compatibilidade implementadas e testadas: - Mesma cor ou vizinhos DIRETOS no espectro → calcula % por proximidade de RGB (40%), Hue (30%), Saturation (15%) e Value (15%). - Cores não-vizinhas ou opostas no espectro → 0% obrigatório. - Neutro vs Colorido → 0% obrigatório. - Branco vs Preto (opostos absolutos de luminosidade) → 0% obrigatório. - Demais neutros (Cinza claro vs Cinza escuro) → calculado por V e RGB, sem floor artificial (pode dar 0% se muito distantes). - Painel inferior exibe thumbnails com miras desenhadas, nomes das cores capturadas abaixo de cada imagem, e % central colorida: verde (>49%), amarelo (1-49%), vermelho (0%). 9. ESTADO ATUAL — O QUE ESTÁ FUNCIONANDO (testado e validado) ─────────────────────────────────────────────────────────────── - Interpretação de cores primárias, secundárias e terciárias com alta fidelidade quando calibrado. - Cutoff espectral de 0% para cores opostas ou distantes no Modo Comparação. - Sistema de calibração branco/preto com compensação de temperatura de cor. - Modo Universal: nomes puros sem adjetivos funcionam corretamente. - Modo Estático: congelamento pós-buffer funciona; não gera oscilação. - Áudio com TTS funcional em todos os pontos de interesse. - Flash independente funcional em dispositivos compatíveis. - Interface responsiva para mobile. 11. ESTADO ATUAL — O QUE PRECISA DE LAPIDAÇÃO (pendente) ───────────────────────────────────────────────────────── MODO NORMAL (não-universal): Atualmente o sistema aplica adjetivos "Claro" e "Escuro" por thresholds fixos de V (V < 28 = Escuro; V > 82 e S < 45 = Claro). Porém, isso não está refinado: um Azul nitidamente escuro ainda pode ser lido como "Azul" sem o adjetivo "Escuro" em determinadas condições de luz ambiente, pois a normalização da régua ou a compensação de temperatura podem empurrar V para fora da faixa de corte. É necessário revisar os thresholds ou aplicar uma lógica relativa à régua de calibração para que os adjetivos sejam mais consistentes em TODAS as cores do espectro. 12. PRÓXIMA FASE — MODO AMPLIADO (novo, ainda não implementado) ────────────────────────────────────────────────────────────────── Criar um quarto modo de operação chamado AMPLIADO (ou outro nome a definir). Este modo expande o classificador com cores nuances que não existem no espectro base: - Bege - Marrom - Creme - Vinho - Azul Marinho - Ocre - Salmon - (outros tecidos/tintas) REGRAS DE NEGÓCIO PARA O MODO AMPLIADO: - Deve ser um classificador SEPARADO (classifyColorExtended), nunca mesclado ao classifyColorWithRuler principal, para preservar a estabilidade do Modo Universal e Modo Normal. - Deve manter adjetivos "Claro" e "Escuro" para TODAS as cores, inclusive as novas. - O sistema precisa distinguir quando um tom desviado de Amarelo vira Bege, quando um Vermelho escuro vira Vinho, e quando um Laranja/Amarelo escuro vira Marrom. Isso é feito por análise combinada de H+S+V, não só de H. - Preocupação do usuário: se a faixa de Hue for muito ampla, Amarelo pode colidir com Bege ou Laranja. A solução é definir janelas espectrais de entrada (Hue ranges) e, dentro delas, sub-classificar por S e V. - Este modo é voltado para análise de tecidos, tintas, design de interiores e outras aplicações onde nuance de cor é mais importante que velocidade. 13.
